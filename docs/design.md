# 設計判断ログ

`proc-lifegame` は、効率の良いライフゲーム実装を目指したものではなく、プロセス管理・IPC・同期制御・Coordinator-Worker構成といったインフラ寄りの技術を観察しやすい題材で学ぶことを目的としています。ここでは、その目的に基づいて行った主要な設計判断を、背景・採用案・却下案の順で記録します。

## 1. 1プロセス = 1セル

- **背景**: ライフゲームの各セルをどの粒度で表現するかは自由に選べる。goroutineでもスレッドでもよい。
- **採用案**: 1つのセルを1つのLinuxプロセス（`proc-cell`）として起動する。`8x8`の盤面なら64プロセス、`40x16`なら640プロセスが同時に立ち上がる。
- **却下案**: goroutineベースの並行実装。プロセス起動・スケジューリングコストが小さく効率は良いが、OS上でプロセスがどう管理されるかを観察するという目的には合わない。
- **補足**: この設計は効率よりも「実際に大量のプロセスがどう管理され、どう同期されるかを観察できること」を優先している。

## 2. TCP localhost + JSON によるプロセス間通信

- **背景**: `proc-coordinator` と各 `proc-cell` の間で、世代ごとに近傍状態と次状態をやり取りする必要がある。
- **採用案**: TCP（`127.0.0.1`）上でJSONメッセージ（[`internal/protocol/protocol.go`](../internal/protocol/protocol.go) の `StepRequest` / `StepResponse`）をやり取りする。
- **却下案**: UNIXドメインソケットや共有メモリ。速度面では有利だが、メッセージ内容が可視化しづらく、観察対象としての見通しが悪くなる。また通信をlocalhostのTCPに閉じておくことで、将来的にプロセスをコンテナへ分離する（Docker化する）構成へ転用しやすい、という拡張性も理由の一つ。
- **補足**: `proc-coordinator` は起動時に空きポート帯を探索し、`proc-cell` ごとに異なるポートを割り当てる（`--base-port` オプション）。

## 3. バリア同期による世代管理

- **背景**: ライフゲームでは、全セルが同じ世代の状態をもとに次世代を計算する必要がある。一部のセルだけが先に次世代へ進んでしまうと、盤面全体の整合性が崩れる。
- **採用案**: `coordinator` が全 `cell` の応答が揃うまで次世代への遷移を待つバリア同期を採用する。

  ```text
  Generation N
    ↓
  coordinator が各 cell に近傍状態を送信
    ↓
  各 cell が次状態を計算
    ↓
  coordinator が全 cell の応答を待つ
    ↓
  全 cell の応答が揃ったら Generation N+1 へ進む
  ```

- **却下案**: 各セルが自分のペースで非同期に世代を進める方式。プロセス間の独立性は高まるが、世代の一貫性が失われ「ライフゲームとして正しい」状態を維持できない。

## 4. UIはcoordinatorのsnapshotのみを購読する

- **背景**: 表示層（`internal/ui`）が盤面状態をどこから取得するかは、直接 `cell` に問い合わせる方式と、`coordinator` を経由する方式のどちらも取り得る。
- **採用案**: UIは `coordinator` が生成した盤面スナップショットだけを参照し、`cell` へ直接通信しない。状態の正本（source of truth）は `proc-cell` が持ち、`proc-coordinator` は同期制御と表示用スナップショットの生成に専念する。
- **却下案**: UIが各 `cell` に直接ポーリングする方式。表示側と各セルが密結合になり、プロセス数が増えるほど接続管理が煩雑になる。
- **補足**: この責務分離により、UI実装（TUI/CUI）を差し替えても `coordinator` 側のプロセス管理・同期ロジックに影響しない。

## 5. `proc-cell` バイナリの解決方法

- **背景**: `proc-coordinator` は起動のたびに `proc-cell` の実行ファイルを見つける必要がある。
- **採用案**: `--cell-bin` で明示指定がなければ、まず自分自身と同じディレクトリにある `proc-cell` バイナリ（sibling）を探し、見つからなければ一度だけ `go build` して一時ディレクトリに生成し、以降のセル起動で使い回す（[`internal/proc/proc.go`](../internal/proc/proc.go) の `resolveCellBinary`）。
- **補足**: `go run` でそのまま起動しても動くよう、ビルド済みバイナリが無い環境をフォールバックとしてカバーしている。
